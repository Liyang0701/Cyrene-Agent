# 验证记录：微信多连接账号

## 自动化验证

- 账号存储、`safeStorage` 加密、旧版迁移与损坏隔离。
- 多账号连接池、绑定者静默过滤、独立停用、重连与删除。
- 复合对话身份、历史/关系/上下文隔离、同对话串行与跨账号公平调度。
- 多账号设置视图、单二维码事务和逐账号命令。
- 显式出站路由、渠道资料/权限隔离、离线任务待投递与删除归档。
- 角色切换会暂停未开始的微信消息，等待在途回复发送完成；切换失败后恢复旧角色队列。

## 真实环境待验收

1. [已通过] 配置 2–3 个微信 iLink 账号，确认至少 2 个同时在线。
   - 2026-07-17 用户实测：账号数量 2，在线数量 2，两个账号均显示“在线”。
2. [已通过] 从两个扫码绑定微信同时发送文字，确认回复账号、上下文和历史不串线。
   - 2026-07-17 用户实测：账号 1 回复“账号一收到”，账号 2 回复“账号二收到”，两边均成功且无串号。
   - 2026-07-17 用户实测：账号 1 的独立暗号为“青鸟七号”，账号 2 为“白鲸九号”；后续询问均只返回本账号暗号，无交叉污染。
3. [自动化通过，真实环境不可执行] 从非绑定联系人发送消息，确认无回复且无 LLM/工具/历史副作用。
   - 2026-07-17 用户尝试从非绑定者发送，但微信端无法建立会话/发送消息，因此无法提供“消息已抵达 iLink 但被 Cyrene 过滤”的真实证据。
4. [已通过] 在一个账号正在生成回复时请求切换角色，再从另一账号发消息；确认旧回复完整发出，并在重启后由新角色处理后续消息。
   - 2026-07-17 用户实测：账号 1 的在途回复因 Qwen HTTP 500 `System message must be at the beginning` 中断；账号 2 切换期间消息在重启后未恢复；两个账号重启后均在线。
   - 角色方向校正：测试前活动角色为星野，重启后为昔涟。
   - 根因 1：OpenAI 适配器会向 Qwen 发送多个 `system` 消息；已改为按原顺序合并，并保证唯一的 `system` 消息位于消息数组首位。
   - 根因 2：角色切换控制重启会丢失内存中的微信排队消息；已增加经 `safeStorage` 加密的持久化入站箱，完成回复后删除，重启后按账号和绑定者重新校验并恢复投递。
   - 修复后自动验证：167 个测试文件、1045 项测试全部通过，主进程/预加载/渲染构建通过，OpenSpec 严格校验通过。
   - 2026-07-17 根因 1 修复后用户实测：昔涟完整回复三段内容，未再出现 Qwen HTTP 500。
   - 2026-07-17 完整复测：账号 1 的昔涟在途回复完整且包含指定结尾；应用自动重启并切换至星野；账号 2 的排队消息在重启后恢复并收到“星野”；两个账号均恢复在线。
5. [已通过] 建立微信渠道提醒后断开原账号，到期后恢复连接，确认只补发一次。
   - 2026-07-17 用户实测：`7712主号` 的提醒创建成功，重连后收到 0 次；`3072` 未误收，两个账号均恢复在线。
   - 根因证据：模型把“3 分钟后”换算成过去的 `2025-04-05`，调度器在断线前立即投递并标记为 `delivered`；历史测试任务也存在 2024/2025 错误年份。
   - 修复：相对提醒改用 `delayMinutes`，由本机时钟计算绝对时间；任务服务拒绝过去时间，避免错误年份被当作立即到期任务。
   - 2026-07-17 修复后用户实测：`7712主号` 的任务准确延迟 179.999 秒；账号离线到期后重连收到 1 次提醒，`3072` 未误收，两个账号均恢复在线。落盘任务最终状态为 `delivered`。
