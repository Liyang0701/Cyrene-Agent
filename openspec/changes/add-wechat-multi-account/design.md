# 设计：微信多连接账号

## 领域边界

- **微信连接账号**由 `ilinkBotId` 唯一标识；备注名只用于展示。
- **微信账号绑定者**由该账号扫码登录返回的 `ilinkUserId` 唯一标识。
- **微信对话身份**由结构化字段 `{ connectionAccountId, contactId }` 表达。初版中 `contactId` 必须等于账号绑定者。
- 所有账号共享全局活动角色、LLM、TTS 和 ASR 服务，但不共享渠道用户资料、历史、关系、记忆、任务或权限。

## 模块 seam

保留 `ChannelManager` 按 `ChannelId` 注册一个 adapter 的接口。微信注册一个 `WechatMultiAccountAdapter` 外观，内部拥有：

- `WechatAccountStore`：账号索引、元数据、加密凭据和旧版迁移。
- `WechatAccountConnectionPool`：逐账号创建、启动、停止、重连和销毁连接运行时。
- `WechatAccountRuntime`：一个账号的一条 iLink client、长轮询、同步游标、context token 和状态。
- `WechatConversationRouter`：绑定者校验、结构化对话身份、入站/出站账号选择。
- `WechatFairScheduler`：同对话串行、跨账号有限并发和公平排队。
- `WechatAccountStatusView`：向 IPC 和设置页提供脱敏汇总及逐账号状态。

连接池对通用渠道层暴露一个小接口，避免让账号生命周期、凭据和协议状态泄漏到 `ChannelManager`。

## 数据模型与存储

账号索引保存非敏感元数据：

```ts
interface WechatConnectionAccountRecord {
  connectionAccountId: string; // ilinkBotId
  label: string;
  enabled: boolean;
  credentialState: "ready" | "login_required" | "corrupt";
  createdAt: string;
  updatedAt: string;
  lastConnectedAt?: string;
}
```

逐账号凭据至少包含 `botToken`、`baseUrl`、`ilinkUserId`，使用 Electron `safeStorage` 加密后独立落盘。单个文件损坏只影响对应账号。文件名使用经过验证的安全键，不直接信任远端 ID 作为任意路径。

账号私有运行状态按账号分开保存：

- `getUpdates` 同步游标；
- `{ accountId, contactId }` 对应的 context token；
- 最近连接和错误摘要；
- 离线待投递任务引用。

对话状态位于活动角色的 Character State Root 内，并以结构化微信对话身份派生稳定 session key。派生算法必须无歧义，不能只使用联系人 ID，也不能依赖可变备注。

## 旧版迁移

启动时检测旧 `weixin/credentials.json`：

1. 读取并验证旧凭据。
2. 以其 `ilinkBotId` 创建第一个账号，默认备注“微信账号 1”并启用。
3. 加密写入新凭据和账号索引。
4. 重新读取并校验新结构。
5. 成功后归档旧文件，不再双写。
6. 失败时保留旧文件并进入单账号兼容模式，同时显示可操作错误。

旧微信历史优先归属该迁移账号；无法证明归属的内容进入“旧版微信历史”归档，不能混入任一新对话。

## 连接生命周期

- 应用启动时并行启动所有已启用且凭据有效的账号。
- 每个账号独立维护长轮询、AbortController、同步游标和重连状态。
- 临时故障采用带随机抖动的指数退避；全池限制同时重连数量。
- 明确凭据失效时停止自动重试，状态变为“需要重新扫码”。
- 停用保留凭据和状态但停止连接。
- 退出登录清除凭据并保留账号条目与归档状态。
- 删除清除账号配置和凭据、停止任务触发，但不自动永久删除历史、记忆和任务归档。

单次只允许一个二维码登录会话。扫码成功后，相同 `ilinkBotId` 刷新既有账号；新 ID 创建新账号。关闭设置页不取消，取消、过期和退出应用会终止待登录会话。

## 入站安全与身份

每个账号在协议解析后、任何副作用之前检查：

```text
message.fromUserId === accountCredential.ilinkUserId
```

不匹配时立即静默丢弃，不写消息日志正文、不广播桌面、不下载媒体、不调用 ASR、LLM、工具、历史或记忆。安全日志只记录账号脱敏标识、事件类型和时间，不记录联系人完整 ID 或正文。

合法入站消息携带 `connectionAccountId` 和结构化对话身份贯穿 dispatcher、历史、记忆、权限、任务与出站流程。桌面显示为“账号备注 / 绑定者”，内部主键不用显示名。

## 出站路由

- 入站回复必须使用消息携带的 `connectionAccountId` 和对应 context token。
- 主动消息、桌面手动发送和任务投递必须显式提供连接账号与绑定者。
- 未提供账号、账号不匹配、账号离线或 context token 不可用时返回明确错误，不选择隐式默认账号。
- 设置页的“常用账号”未来只能作为界面预选，不改变后台路由要求。

## 并发与公平性

- 每个微信对话身份一条串行队列。
- 全局 LLM 并发初版为 2。
- 不同账号按轮转方式获得执行机会，避免单账号饿死其他账号。
- 同一对话的新消息排队，不取消、不擅自合并正在处理的消息。
- TTS、媒体上传和发送保持原对话身份，但可与其他对话的 LLM 阶段并行。
- 单请求超时或失败只结束对应任务。

## 角色切换协调

所有账号使用全局活动角色。角色切换开始后：

- 微信连接和长轮询保持在线；
- 新消息进入暂存队列，不开始角色绑定工作；
- 已开始的回复使用旧角色完成 LLM、TTS 和发送；
- 切换成功后，未开始消息使用新角色；
- 切换失败则继续旧角色；
- 账号级启停和重连不触发角色切换。

## 渠道资料、权限和任务

- 每个微信对话身份拥有独立 Channel User Profile，不读取或继承本机 Global User Profile。
- 每个账号拥有独立 Channel Account Permission Policy；默认禁止本机文件、Shell、邮件发送和其他高风险能力，只开放明确批准的低风险工具。
- 即使扫码者与本机所有者是同一个人，也不自动继承桌面权限。
- 微信创建的 Channel User Task 归属于原微信对话身份；到期只通过原账号投递。
- 账号离线时任务变为待投递，恢复后最多补发一次。
- 账号删除后未完成任务归档并停止触发，不迁移到其他账号。

## 设置页和状态

顶部显示账号汇总，例如“3 个在线，1 个需要重新登录”，并提供“添加微信账号”。

每个账号显示备注、脱敏 Bot ID、状态、最近成功连接时间和最近错误摘要，支持：

- 修改备注；
- 启用/停用；
- 立即重连；
- 重新扫码；
- 退出登录；
- 删除账号。

不提供联系人审批、“所有联系人可用”或“一键全部退出”。处理中和排队数量按账号展示。

## 可观察性与隐私

- 所有连接、调度、错误和投递日志带账号备注与脱敏账号 ID。
- 不记录 Token、二维码、完整 `ilinkUserId`、context token 或未授权消息正文。
- 普通设置导出、角色包、诊断包和跨设备迁移均排除凭据。
- 状态汇总必须区分在线、连接中、重连中、已停用、需要扫码和错误。

## 验证策略

1. 单元测试：账号存储、safeStorage、ID 安全、迁移、绑定者过滤、复合路由、调度公平性和状态聚合。
2. 集成测试：两个 mock iLink 账号并行长轮询、独立断线/恢复、出站账号选择和 context token 隔离。
3. 状态测试：角色切换、离线任务、删除/退出、二维码互斥。
4. UI 测试：账号列表、状态、逐账号操作和错误恢复。
5. 真实验收：3 个账号配置、2 个账号同时在线，从两个绑定微信分别发送消息并确认回复与记忆不串线。
