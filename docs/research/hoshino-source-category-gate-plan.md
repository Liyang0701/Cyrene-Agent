# 星野语料库：补齐来源类别 Gate 的一手来源采集计划

## 研究范围与结论

- 研究日期：2026-07-18
- 研究阶段：`/research`
- 研究范围：只规划小鸟游星野的高可信一手来源采集；研究阶段本身不改动人格包、运行时代码或既有语料。后续执行记录与研究结论分开保留。
- 结论：现有 Corpus 不是“缺更多主线”，而是缺三种**已经被验证器支持、但尚无已复核条目**的来源类别。以“官网试听 + 官方公告 + 日服客户端语音”三条路线即可补齐类别 Gate；其中客户端语音还应一次性补足日文数量 Gate。个人/羁绊剧情在技术上也能作为第三类来源，但受絆等级解锁约束，适合作为随后优先的质量增强或客户端语音不足时的替代路线。

## 优先采集路线

### P0（无需客户端）：日服官网角色页与官方试听

- **拟计类别**：`official-site`。
- **官方 URL**：[日服官网角色页](https://bluearchive.jp/character)；其当前页面加载的星野试听资源为 [官方 WAV](https://webusstatic.yo-star.com/bluearchive_jp_web/media/02.73ff260c.wav)。
- **可核验事实**：页面组件直接列出 `ホシノ`、`CV:花守ゆみり`、`学園:アビドス高等学校`、`学年:3年`、生日和身高；试听提供一条约 2.743 秒的官方日文日常语音。现有研究把该试听的文本候选记为「適当に、適当に」，但正式入库前仍须重新听取原始 WAV 并以音频为准。
- **可支持的星野特征**：日常状态的轻缓重复、不过度用力的节奏；同时以角色资料锚定身份事实。它不能单独推导“任何场合都懒散”。
- **建议产出**：至少 2 条独立记录：1 条 `official-fact`（保证类别锚点），1 条经人工听辨确认的 `official-dialogue`（预计贡献 1 条日文）。
- **是否需用户在客户端手动录制**：否。
- **执行边界**：保存当日下载的原始 WAV 与必要页面资料到 Git 外证据目录、记录 SHA-256 和抓取日期；不从二手文字稿抄录，不以 ASR 直接定稿。

### P1（无需客户端）：日服官方公告中的受限官方事实

- **拟计类别**：`official-publication`。
- **官方 URL**：[官方公告页面](https://bluearchive.jp/news/newsJump/467)；该页的日服官方站数据接口为 [公告 467 JSON](https://api-web.bluearchive.jp/api/news/detail?id=467)。
- **可核验事实**：公告标题为“ピックアップ募集紹介：★3「ホシノ」、★3「ヒナ」”，正文将她称为“アビドス最強の「ホシノ」さん”。这是官方宣传口径，适合记录为“战斗声望/官方表述”，不应扩写成全领域万能或日常人格结论。
- **可支持的星野特征**：战斗定位与官方宣传事实的边界；可防止角色包把“强”误写为无条件压迫感。
- **建议产出**：至少 1 条 `official-fact`，用于使 `official-publication` 真正进入类别计数；**预计贡献 0 条日文对话数量**，因为它不是角色实际对白。
- **是否需用户在客户端手动录制**：否。
- **执行边界**：保存页面/API 返回的原始内容或稳定截图并记录 SHA-256；只摘录支撑结论所需的短语，不复制整篇公告。

### P2（类别与数量主路线，需日服客户端）：星野语音库

- **拟计类别**：`in-game-voice`。
- **官方 URL**：[《Blue Archive》日服官网](https://bluearchive.jp/)（客户端与版本的官方来源）；公开网页没有可稳定深链到每条游戏内语音的 URL，因此该 URL 只用于版本溯源，最终证据必须是日服客户端画面和音频本身。
- **可核验内容**：星野的主页/登录/编队/强化/礼物/战斗开始/EX 技能/胜利等语音。每条要同时保留星野身份、语音条目名称或可辨别 UI、实际播放音频与完整句子。
- **可支持的星野特征**：日常对老师的称呼和松弛节奏、在事务或战斗中变得简洁的表达、照顾/提醒与轻微调侃；比主线高压场景更能校准默认语体。
- **建议产出**：从互不重复的菜单项挑选 **至少 17 条** 日文 `official-dialogue`。优先覆盖：登录/主页互动、好感或礼物、编队/强化、战斗开始、技能、胜利/失败；不要把同一句口癖拆成多条来凑数。
- **数量效果**：即使 P0 的官网试听最终不适合计数，17 条客户端语音也会使日文达到 `151/150`；若 P0 试听通过则为 `152/150`，留出一条复核淘汰缓冲。
- **是否需用户在客户端手动录制**：是。用户此前已说明日服可在手机或平板打开，故不要求在 Mac 上运行游戏。
- **执行边界**：录制一个连续的屏幕视频即可，但每次切换语音条目都应停留足够久，让 UI 标签与完整语音都可辨认。录制后由项目侧裁剪为可复核片段、保存 Git 外原件、计算哈希，并以人工“画面 + 听辨”确定文字；OCR/ASR 只能做定位辅助。

### P3（建议紧随 P2，需日服客户端）：星野个人/羁绊剧情

- **拟计类别**：`in-game-relationship`（可作为第三个缺失类别，也可在 P2 后成为第六类）。
- **官方 URL**：[日服 2024-06-26 更新公告](https://bluearchive.jp/news/newsJump/460)（[官方数据接口](https://api-web.bluearchive.jp/api/news/detail?id=460)）明确：普通 ★3「ホシノ」新增 1 话羁绊剧情，絆ランク 15 解锁；最终来源仍是日服客户端内该“絆ストーリー”画面。
- **可核验内容**：已解锁的关系剧情中，星野与老师的日常邀请、关心、边界和信任变化。公告只证明该普通星野剧情及解锁条件存在；实际台词和完整上下文必须以用户账号画面为准，不能在采集前凭 wiki 或记忆预写。
- **可支持的星野特征**：这是“对老师默认较亲密但不擅自宣称原作确认恋爱”的最佳一手边界证据，也能补足日常场景而非重复创伤主线。
- **建议产出**：若已解锁，采集 2～3 段完整小节，预计筛出 10～16 条高信息日文候选；正式数量以逐页复核后为准。
- **是否需用户在客户端手动录制**：是；若当前未解锁，不为本阶段强行刷好感或改变账号进度。
- **执行边界**：它是 P2 后的质量增强；若 `in-game-voice` 在用户账号中不可采集，而絆等级 15 已解锁，它也可作为第三个缺失类别。但届时必须实际筛得至少 16 条额外日文，不能事先假定一话剧情一定够数。

### P4（可选备用，不增加类别）：既有日服官方视频

- **现有类别**：`official-video`，因此不能解决三类缺口。
- **官方 URL**：[《先生、ちょっと時間もらうよ～…》#2](https://www.youtube.com/watch?v=HblzE-T6mfw)、[第 3 章官方预告](https://www.youtube.com/watch?v=ZgVi0dIziy8)。
- **用途**：若 P2 中少于 17 条可用语音，可再从官方视频直接听辨少量、上下文明确的短句补日文数量；不代替客户端语音类别。
- **是否需用户在客户端手动录制**：否。

## 最小闭环与验收

按 P0 → P1 → P2 执行时，最小通过式为：

```text
既有类别 2
+ P0 的 1 条已复核 official-site 记录
+ P1 的 1 条已复核 official-publication 记录
+ P2 的 1 条以上已复核 in-game-voice 记录
= 5 类

既有已验证日文 134
+ P0 试听（若通过，1 条）
+ P2 至少 17 条互不重复的客户端语音
= 151～152 条日文
```

每条正式进入 Corpus 前都必须满足：

1. 来源是上表对应的官方网页、官方公告或日服客户端直接画面；不把来源表中的 URL 当作已验证条目。
2. Git 外原件、来源 SHA-256、采集日期、服务器、角色形态、短而可复核的定位信息齐全。
3. `in-game-voice` 与 `in-game-relationship` 使用 `in-game-capture` 定位，包含菜单/章节与时间点；音频和画面人工复核通过。
4. 官网/公告来源与 `entries.jsonl` 的 `sourceHash` 一致；公告保持 `official-fact`，不冒充角色对白。
5. 运行 `npm run corpus:validate -- character-corpora/blue-archive.hoshino/main-story`，预期不再出现日文数量和来源类别阈值问题；再运行相关测试与 `git diff --check`。

## 证据边界、复用与不引入项

### 明确不采用的材料

- wiki、攻略站、角色台词整理页：只能帮助私下定位，不能进 Gate。
- 非官方转载、B 站二创、烧录字幕、字幕组翻译：不作为本计划的任何来源；既有国服纯游戏录屏镜像的项目级例外也不扩展到本轮日文证据。
- 自动转写、OCR、社区翻译：只帮助定位，不能替代原始画面/音频的人工复核。

### 复用什么

- 复用仓库已有的 Character Corpus 清单、六种来源类别、`in-game-capture` 定位规则、哈希绑定和 `corpus:validate`，不需要改验证器。
- 复用当前 Git 外证据目录与既有的“保留原件 → 记录哈希 → 裁剪定位 → 人工复核 → 再入库”流程。
- 复用现有本地媒体工具做裁剪、查看和哈希；其工作只服务保存与复核，不替代证据判断。

### 不自研、也不整套接入其他项目的原因

- 不自研网页爬虫、游戏资源解包器、语音自动对齐器或第三方台词数据库：本轮需要的规模只有官网 1 条、公告 1 条和约 17 条客户端语音，复杂自动化不能提高可信度，反而会模糊版本、授权和说话者边界。
- 不整套接入 wiki、游戏资料站或其他角色语料项目：它们通常混合转录、自译、版本差异和二创补写，无法满足本仓库对“来源哈希 + 人工复核 + 可追溯定位”的 Gate。
- 这不是否定成熟工具的价值：媒体裁剪和哈希继续复用成熟本地工具；可信度判定仍由当前 Corpus 规则和可直接核验的官方原件负责。

## 风险与后续决策

- 官网是 SPA，CDN 资产可能随站点更新替换；采集时必须保存原件与哈希，不能只依赖 URL。
- 官方公告的宣传语是事实类证据，不应挪作“星野会如何对用户说话”的模仿样本。
- P2 是否可一次取得 17 条，取决于用户账号是否持有普通星野以及语音菜单可见范围；如果不足，先从同一客户端的其他可见语音补足，不用非官方录音替代。
- P3 若关系剧情未解锁，则记录为后续事项；不影响 P0/P1/P2 达成最低 Gate。
